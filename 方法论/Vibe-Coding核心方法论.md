# Vibe Coding 核心方法论

## 高级方法论：Plan Mode（先谋后动）

### 原理
在 AI 动手前，先开启"计划模式"。要求它输出一个 Checklist。

### Prompt 模板
```
我想实现一个用户登录功能。先不要写代码，请列出你需要创建的文件清单和核心逻辑步骤。
```

### 价值
- 降低试错成本：大型重构如果方向错误，推倒重来的代价很高
- 保持人的掌控感：开发者始终知道 AI 要做什么
- 延长任务连续运行时间：有了计划作为"导航"，AI 不容易跑偏

## Micro-commits（微提交）

### 原则
每当 AI 完成一个小的子功能（比如写完一个按钮组件），立即进行 Git Commit。

### 技巧
如果 Vibe 崩了（代码报错跑不通），直接 `git reset --hard`，重新调整你的描述。

### 流程
```
AI 写代码 → 自动测试 → Git commit → 继续下一步
```

## Agent Audit（影子复核）

### 流程
不要只信一个模型：
1. 用 **Claude** 写核心代码
2. 用 **GPT-5** 检查 UI 样式
3. 用 **Gemini** 扫描是否存在安全漏洞

### 核心武器：.rules 约束文件

在 Vibe Coding 中，如何防止 AI "放飞自我"？你需要一个通用的规则文件。

### 为什么需要它？

1. **统一风格**：强制 AI 使用特定的库（如：必须用 Lucide 图标，不能用 FontAwesome）
2. **防御编程**：要求 AI 必须处理异常，不能只写"快乐路径"
3. **减少沟通成本**：不需要每次都告诉 AI "不要写注释"或者"用中文回复"

### AI 开发代理通用规则模板

```markdown
# AI 开发代理通用规则 (Standard Vibe Rules)

## 1. 身份与角色
- 你是一个资深的 Full-stack Engineer，倾向于简洁、高性能、可维护的代码。
- 在修改代码前，必须先简要说明你的"修改思路"。

## 2. 技术栈约束
- 前端：React 19, Tailwind CSS, TypeScript (Strict Mode)。
- 状态管理：优先使用 Zustand，避免过度使用 Redux。
- 图标库：仅使用 Lucide-React。

## 3. 编码规范
- **DRY 原则**：不要重复逻辑，公共逻辑必须抽离到 /hooks 或 /utils。
- **错误处理**：所有 API 调用必须包裹 try-catch，并使用自定义的 ErrorBoundary。
- **类型安全**：禁止使用 `any`，必须定义 Interface 或 Type。
- **样式**：优先使用 Tailwind 类名，禁止写行内 style。

## 4. 交互流程
- 修改文件前，先扫描全局上下文，确保不会破坏其他模块。
- 如果需求模糊，先提问，不要猜测。
- 每次完成功能后，自动编写一个简单的 Vitest 测试用例。

## 5. 禁止行为
- 禁止删除用户原有的注释。
- 禁止引入未经说明的第三方大库。
- 禁止在未处理 Edge Cases 的情况下关闭任务。
```

### 工具不同，规则文件名称不同

| 工具 | 规则文件 |
|------|---------|
| Cursor | `.cursorrules` |
| Claude Code | `AGENT_RULES.md` |
| 通用 | `.ai-rules` |

## Vibe Coding 时代的生存法则

Vibe Coding 并不是让程序员失业，而是将程序员从"搬砖工"解放为"架构师"。

未来的核心竞争力不再是记住多少 API，而是：

1. **拆解问题的能力**
   - 你能否将复杂需求拆解成 AI 能听懂的小模块？

2. **审美与判断力**
   - AI 给了你三个方案，你能一眼看出哪个是"优雅"的吗？

3. **规则制定能力**
   - 你是否有一套属于自己的 `.rules` 武器库？

## 关键心法：只要 Vibe 不乱，代码随叫随到

### 避免上下文偏移

当项目超过 50 个文件后，AI 可能会"忘记"你在初始化时定下的全局变量规范，开始胡乱造轮子。

**解决方案**：
- 使用 `.ai-rules` 固化全局规范
- 每次重要对话开始时，让 AI 先读取项目文档

### 避免代码冗余

如果不加约束，AI 倾向于写出最稳妥但最啰嗦的代码，导致项目迅速变重。

**解决方案**：
- 在规则中明确要求"简洁优先"
- 定期让 AI 审查并重构冗余代码

### 避免"幻觉"陷阱

AI 有时会自信满满地调用一个根本不存在的库函数。

**解决方案**：
- 让 AI 先列出要使用的库，确认后再写代码
- 使用 Agent Audit 让多个模型交叉验证
